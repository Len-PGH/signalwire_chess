Architecture Deep-Dive

♟ SignalWire Chess — How It Works

An interactive chess board wired to a live SignalWire AI voice agent. You play by speaking moves or by moving the pieces on screen, over a real-time WebRTC call — and both paths stay in sync with the AI opponent. This page explains every moving part and cites the exact SDK functions each one relies on.

1System overview

There are five subsystems. The browser holds the visible board and the WebRTC media; the SignalWire cloud runs the AI voice turn; the Python app is the single source of truth for the game and hosts both the AI's function endpoints and a small REST API for the board.

The single most important fact about the design: the browser and the AI never talk directly. The AI runs in the SignalWire cloud; the browser talks to it only through the media call and cloud-delivered events. Everything that has to be shared — the board — lives in the Python app, and both sides read/write it there. Section 2 explains why that shapes the whole system.

2The core challenge

The obvious design — "when the user clicks a move, tell the AI" — is not directly possible. The browser SDK (@signalwire/js) can receive application events from the AI (user_event) and can send audio and DTMF, but it has no channel that injects a fact into the AI's live voice turn.3 The AI decides what to do based on what it hears and what its functions return.2

So how does a silent, on-screen click reach a voice AI?

We invert it. Clicks are written to a shared store on the server. The AI is configured to call a "did anything happen?" function during the natural silences in the conversation, and that function reports any moves the AI hasn't acknowledged yet. The board is the shared truth; the AI polls it.

Two SignalWire capabilities make the polling loop work, both set on the SWAIG function results:

  • enable_functions_on_timeout(True) — permits the agent to call functions when the speaker times out (i.e. the user goes quiet) instead of only after speech.12
  • wait_for_user(timeout=6) — tells the agent to yield and wait, arming the next silence timeout.1

The loop is bootstrapped from the very first turn by set_params({"functions_on_speaker_timeout": True}) on the agent, so polling is armed before the user has said anything.2

3The AI agent — SWML & SWAIG

The opponent is a Python class built on the SignalWire AI Agents SDK.1 Subclassing AgentBase generates the SWML document (the ai verb, prompt, language, hints) and exposes each @self.tool-decorated method as a SWAIG function the LLM can call.12 The server is an AgentServer that mounts the agent at /chess and also serves the web client and REST API.

app.py — a voice move

@self.tool(name="make_move",
  description="Make the player's chess move; the engine replies automatically.",
  parameters={"type":"object","properties":{"move":{"type":"string"}},"required":["move"]})
def make_move(args, raw_data):
    gid, state = _state_for(raw_data)
    ok, msg, payload = do_player_then_engine(state, args["move"], source="voice", game_id=gid)
    result = SwaigFunctionResult(msg)          # the text the AI will speak
    result.swml_user_event(payload)          # push board_update → browser
    result.enable_functions_on_timeout(True) # keep silence-polling armed
    return result

A SwaigFunctionResult carries three things back to the cloud AI turn:1 the natural-language response the model speaks, and a list of actions — here swml_user_event(payload), which emits a SWML user_event that is delivered to the browser over the call,12 and enable_functions_on_timeout. The payload is the same board_update dictionary the REST API returns, so the browser has one render path regardless of who moved.

The polling function

check_board_moves is the function the AI calls during silence. It reads any moves recorded since it last looked, marks them acknowledged, and — if there were any — narrates them and pushes a fresh board_update. Either way it re-arms the loop.

app.py — catching clicked moves

def check_board_moves(args, raw_data):
    gid, state = _state_for(raw_data)
    rows = db_unnarrated(gid)                     # moves the AI hasn't voiced yet
    if not rows:
        r = SwaigFunctionResult("No new board moves. Stay quiet and keep waiting.")
        r.enable_functions_on_timeout(True); r.wait_for_user(timeout=6)
        return r
    db_mark_narrated([row[0] for row in rows])
    r = SwaigFunctionResult(f"The player moved: {desc}. Acknowledge it, then wait.")
    r.swml_user_event(board_payload(state, board))
    r.enable_functions_on_timeout(True); r.wait_for_user(timeout=6)
    return r

Other tools follow the same shape: new_game, change_difficulty, resign, get_board (resync/describe), and explain_term (a chess glossary). Each returns a spoken response and, where the board changed, a user_event.

4The browser client

The front end uses the @signalwire/js v4 browser SDK.3 It fetches a token from the app, creates a client, and dials the agent's address. Note it identifies the game to the AI by passing userVariables on dial() — that game_id flows into every SWAIG call's raw_data, which is how the server-side functions know which game to act on.

web/app.js — connecting and subscribing

const client = await SignalWire({ token });
call = await client.dial(destination, {
  audio: true, video: false, receiveAudio: true, receiveVideo: true,
  userVariables: { game_id: gameId, interface: 'chess-web' }   // → raw_data server-side
});

// the AI's board_update events arrive here; one render path for voice + click
call.subscribe('user_event').subscribe(evt => {
  const p = evt?.params ?? evt;
  if (p?.type === 'board_update') renderBoard(p);
});
call.remoteStream$.subscribe(stream => attachRemoteStream(stream));  // Sigmond's avatar
call.status$.subscribe(st => /* connected / disconnected UI */);
  • SignalWire({ token }) creates the client; client.dial(...) starts the WebRTC call.3
  • call.subscribe('user_event') is the receive side of swml_user_event — every board change from the AI lands here.31
  • call.remoteStream$ exposes the agent's audio+video (the avatar in the "black window").3
  • A clicked move is sent to the REST API (POST /chess/move); the response body is the same board_update shape, so the clicking browser updates instantly without waiting on the AI.

5Rules & engine

All chess correctness is delegated to python-chess,4 and the opponent's moves come from Stockfish.5 The app never trusts a move blindly: text (from voice or a click) is parsed to a legal move, illegal input is rejected with a helpful list, and captures/checks/mate are all derived from the library.

app.py — the opponent's reply

with chess.engine.SimpleEngine.popen_uci(STOCKFISH_PATH) as eng:
    eng.configure({"Skill Level": skill})            # 0–20, per difficulty
    return eng.play(board, chess.engine.Limit(time=movetime)).move
  • Move parsing accepts UCI (e2e4), SAN (Nf3, O-O), and loose natural language ("pawn to e4") via board.parse_san / chess.Move.from_uci and a normalizer.4
  • Legality is move in board.legal_moves; state transitions use board.push(); endings use board.is_checkmate() / is_stalemate() / result().4
  • Difficulty maps to Stockfish's UCI Skill Level option plus a per-level think time — from beginner (level 0, 50 ms) to master (level 20, 800 ms).5 A random-legal fallback keeps the game playable if the engine binary is missing.

6Keeping the board in sync

One function, do_player_then_engine(), is the shared core for both input paths: it applies the human move, gets Stockfish's reply, records both to the store, and returns a board_update payload. The source ("voice" vs "click") controls one thing — whether the move is logged as already-narrated.

A spoken move

You say "knight f3" → the AI calls make_move
↓
server applies your move + Stockfish reply, logs both as narrated
↓
swml_user_event(board_update) → browser renders; the AI speaks the reply

A clicked move

You drag a piece → POST /chess/move
↓
server applies your move + Stockfish reply, logs them un-narrated (source=click)
↓
REST response renders your browser immediately
↓  (on the next silence)
the AI calls check_board_moves, sees the un-narrated moves, and voices them

Why "narrated" matters

Voice moves are logged narrated=True because the AI is already speaking them the instant it calls make_move. Click moves are logged narrated=False so the polling function knows they still need to be caught up on out loud. This flag is what stops the AI from either double-announcing or silently missing a move.

7Animated move GIFs

For each turn the server renders an animated GIF — the piece sliding to its square, captured pieces fading, the castling rook sliding too, and the final position held with a check highlight — and the browser overlays it on the avatar window, then fades back. Rendering is done with Pillow,6 drawing Unicode chess glyphs frame by frame and assembling them into a single GIF.

chessgif.py — assembling the animation

conv = [f.convert("RGB").convert("P", palette=Image.ADAPTIVE, colors=48) for f in frames]
durations = [45] * (len(conv)-1) + [1500]        # hold the final position
conv[0].save(buf, format="GIF", save_all=True, append_images=conv[1:],
             duration=durations, disposal=2, optimize=True)  # no loop → plays once

The frames use ImageDraw.text(..., anchor="mm", stroke_width=…) for outlined pieces and Image.alpha_composite for the capture fade; the multi-frame GIF is written with save_all, append_images, per-frame duration, and disposal=2.6

Rendering is deliberately off the move path

A full render takes ~400 ms. Originally that ran inside make_move//chess/move, which delayed the board_update event and left the on-screen board lagging behind a spoken move. The fix: a move only records a lightweight render spec (start position + the moves + orientation) and stamps a gif_id. The actual pixels are rendered lazily on first GET /chess/gif/{id} and cached — so the move response, and the event the board depends on, stay instant.

Verified: /chess/move ≈ 0.44 s (Stockfish only), first GIF GET ≈ 0.54 s (in the browser's own image fetch), cached GET ≈ 4 ms.

8State & persistence

Two layers of state:

  • Live authority — an in-memory GAMES{} dict keyed by game_id, holding the FEN, captured trays, history, and difficulty. Both the AI functions and the REST API mutate this same object, guarded by a lock. This is the truth the board renders from.
  • Move log — a SQLite database in WAL mode7 recording every move UUID-stamped with its source (voice/click/engine), SAN/UCI, resulting FEN, capture, and a narrated flag. This is what check_board_moves queries, and what powers GET /chess/stats.

The SQLite file is stored on a Docker named volume, so the move log and statistics survive rebuilds and restarts.

9Engineering decisions

DecisionWhy
Single web worker (--workers 1)The in-memory GAMES{} store is the shared authority for the voice and click paths; multiple workers would each hold a divergent copy. One worker keeps a single truth.
Poll for clicks instead of pushingNo browser→live-AI channel exists.3 Polling on silence via functions_on_speaker_timeout is the supported way for the AI to observe out-of-band state.2
narrated flag on every moveDistinguishes "the AI already said this" (voice) from "the AI still needs to catch up" (click), preventing double- or non-announcement.
Lazy GIF renderingKeeps ~400 ms of Pillow work off the move/response path so the board_update event — and the on-screen board — never lags the spoken move.
SQLite in WAL modeConcurrent readers (the poll query) don't block the writer (move recording).7
Sync FastAPI endpointsThe SDK's server is FastAPI/Starlette; sync route handlers run in a threadpool,8 so short blocking work (a Stockfish call) doesn't stall the event loop.

The board_update event / payload

The one shape every path produces — pushed as a user_event by the AI and returned as the REST body by the board API. Key fields: fen, last_move, turn, legal_moves, in_check, game_over, result, captured_by_white/black, material, history, player_color, difficulty, and gif_id/gif_url.
